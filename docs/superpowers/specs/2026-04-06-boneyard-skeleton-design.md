# Boneyard Skeleton Loading Design

## Goal

Add pixel-perfect skeleton loading screens to all data-loading pages in LimeL8 using [boneyard-js](https://github.com/0xGF/boneyard). The app currently has zero loading states. Every page that fetches data server-side will show a skeleton while streaming in, using Next.js App Router's built-in `loading.tsx` mechanism. The Messages conversation page (client component) gets an inline loading state.

## Architecture

**Why `loading.tsx` + Boneyard:**
All pages in `app/(main)/` are server components that fetch data at render time. Next.js App Router automatically shows `loading.tsx` while a page's server component is executing. This means:
- Zero changes to existing page files
- Skeletons activate automatically on navigation
- Boneyard `<Skeleton>` components wrap the actual repeating card/list components to capture their real DOM layout

**Boneyard workflow:**
1. Install `boneyard-js`
2. Create skeleton wrapper components for each repeating unit
3. Add `loading.tsx` files that render those skeleton wrappers
4. Import the generated registry in `app/layout.tsx`
5. Run `npx boneyard-js build` against the dev server to generate bone descriptors

## Skeleton Units

Six repeating UI units get Boneyard `<Skeleton>` wrappers. Each wrapper renders the real component when `loading={false}` and the auto-generated skeleton when `loading={true}`.

| Skeleton name | Real component | Used in |
|---|---|---|
| `post-card` | `PostCard` | Feed |
| `listing-card` | `ListingCard` | Marketplace, Listings |
| `org-card` | `OrgCard` | Orgs dashboard |
| `profile-card` | `ProfileCard` | Search/Discover |
| `conversation-item` | Conversation row in `ConversationList` | Messages list |
| `booking-card` | Booking row in `BookingList` | Org dashboard |

## Skeleton Component Pattern

Each skeleton component is a thin wrapper:

```tsx
// components/skeletons/PostCardSkeleton.tsx
import { Skeleton } from 'boneyard-js/react';
import { PostCard } from '@/components/feed/PostCard';

type Props = {
  loading: boolean;
  post?: React.ComponentProps<typeof PostCard>['post'];
};

export function PostCardSkeleton({ loading, post }: Props) {
  return (
    <Skeleton name="post-card" loading={loading} color="#27272a" animate>
      {post && <PostCard post={post} />}
    </Skeleton>
  );
}
```

Key decisions:
- `color="#27272a"` — matches `var(--card)` from the app's dark zinc design system
- `animate={true}` — pulse shimmer enabled
- Child only rendered when data exists (avoids rendering with undefined props)

## Loading Page Pattern

Each `loading.tsx` renders a plausible number of skeleton items matching the page's real content:

```tsx
// app/(main)/feed/loading.tsx
import { PostCardSkeleton } from '@/components/skeletons/PostCardSkeleton';

export default function FeedLoading() {
  return (
    <main className="mx-auto max-w-2xl px-4 py-6 space-y-4">
      {Array.from({ length: 5 }).map((_, i) => (
        <PostCardSkeleton key={i} loading={true} />
      ))}
    </main>
  );
}
```

Count guidelines:
- Feed: 5 post skeletons
- Marketplace/Listings: 8 listing card skeletons (2×4 grid)
- Orgs: 3 org card skeletons
- Messages list: 6 conversation skeletons
- Search: 6 profile card skeletons
- Profile page: banner + avatar placeholder + 3 post skeletons
- Org dashboard: details placeholder + 2 service rows + 3 booking skeletons
- Public org page: banner + avatar + 2 service rows
- Listing detail: image placeholder + info block
- Profile settings: form placeholder

## Messages Conversation (Client Component)

`app/(main)/messages/[profileId]/page.tsx` fetches messages client-side via `useEffect`. Add a loading state:

```tsx
const [isLoading, setIsLoading] = useState(true);
// existing fetch sets isLoading(false) when complete

if (isLoading) return <MessageListSkeleton />;
```

A `MessageListSkeleton` renders 6 alternating left/right message bubble placeholders using a simple inline div approach (not Boneyard, since the bubbles are too dynamic for snapshot capture).

## Boneyard Build Step

After implementation, run once against the local dev server:

```bash
npx boneyard-js build http://localhost:3000
```

This generates `bones/registry.js` (and individual `.bones.json` files per skeleton name). The registry is then imported once in `app/layout.tsx`.

The build step must be re-run if any wrapped component's layout changes significantly.
